### Filter Hierarchy
1. **Default**: User's primary location + radius
2. **Saved locations**: Quick switch between saved cities
3. **Global**: "Search all locations" option
4. **Temporary**: "Search near [City]" for one-time lookups

## User Flows

### New User (First Visit)
1. Lands on homepage
2. Prompt: "Where do you play? [Detect Location] or [Enter Zip]"
3. Sets Phoenix, 25 miles
4. Homepage instantly shows Phoenix content
5. Can browse/change anytime via location selector

### Existing User (Returning)
1. Lands on homepage
2. Automatically sees Phoenix content (stored preference)
3. Location bar shows: "ğŸ“ Phoenix, AZ (25mi)"
4. Can switch to Scottsdale (saved location) with one click

### User Moving to New City
1. Goes to Settings â†’ Locations
2. Changes primary from Phoenix â†’ Denver
3. Entire platform updates to Denver content
4. Can save Phoenix as secondary location

### User Traveling
1. Clicks location selector
2. "I'm traveling to..." â†’ Enters San Diego
3. Temporarily shows San Diego content
4. Can save for future trips
5. Returns to Phoenix when trip ends

Bottom Line
Don't build a separate local feed. Instead:

Make location the default filter on EVERY page
Always show location context (header bar)
Make it stupid-easy to change location
Support multiple saved locations
Default radius to ~25 miles for most users
This keeps the architecture simple while ensuring users never see irrelevant distant content. The platform feels local without needing separate feeds.
